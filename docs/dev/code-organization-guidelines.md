## 1. 文档目标

本文档定义 QuantFlow 的代码组织、文件大小、复杂度、命名、类型、测试和工程质量规范，适用于前端、后端、脚本、测试和后续自动化工具。

本规范服务于 QuantFlow MVP 边界：MVP 只做策略信号、策略广场、策略详情、信号中心、模拟盘、会员订阅和管理后台，不做真实交易所 API 连接、真实下单、半自动交易或全自动交易。

## 2. 文件大小与复杂度限制

### 2.1 文件行数

默认限制：

1. 单个源码文件目标不超过 300 行。
2. 单个源码文件超过 400 行必须拆分。
3. React 页面文件建议不超过 250 行。
4. React 组件文件建议不超过 200 行。
5. 自定义 hook 文件建议不超过 150 行。
6. 后端 Controller 文件建议不超过 200 行。
7. 后端 Service 文件建议不超过 300 行。
8. 单个测试文件建议不超过 300 行。

允许例外：

1. 数据库迁移文件。
2. 自动生成文件。
3. 大型类型声明文件。
4. 静态常量字典。
5. API schema 或 OpenAPI 生成产物。

例外文件必须尽量放在 `generated`、`migrations`、`schemas` 或 `constants` 等清晰目录中，不得把业务逻辑混入例外文件。

### 2.2 函数与复杂度

限制：

1. 单个函数目标不超过 60 行。
2. 单个函数超过 80 行必须拆分或说明原因。
3. 嵌套层级不超过 3 层。
4. 单个函数参数建议不超过 5 个，超过时使用对象参数。
5. 复杂条件必须提取为具名函数或领域规则。
6. 禁止把多个业务流程写进一个函数。

### 2.3 模块拆分触发条件

出现以下情况必须拆分：

1. 一个文件同时包含 UI、请求、权限判断和业务规则。
2. 一个组件同时处理列表、详情、表单和弹窗。
3. 一个 Service 同时处理多个业务域。
4. 一个函数同时读写数据库、调用外部服务、处理权限和拼装响应。
5. 同一逻辑被复制 3 次以上。

## 3. 目录与模块组织

### 3.1 前端组织

推荐按业务域组织：

```text
src/
  features/
    strategies/
      components/
      hooks/
      api/
      types/
      utils/
    signals/
    paper-trading/
```

规则：

1. 页面只做布局和模块组合。
2. 业务组件承载展示逻辑，不直接散落 API 请求。
3. 请求逻辑放在 `api` 或统一 client 中。
4. 复用状态放在 hook 中。
5. 格式化、金额、百分比、日期、风险等级映射放在 `lib/formatters` 或 `utils`。
6. 权限判断放在 `lib/permissions`，不得在页面中重复硬编码。

### 3.2 后端组织

推荐按业务域组织：

```text
src/
  modules/
    strategy/
      domain/
        entities/
        value-objects/
        services/
        events/
      application/
        use-cases/
        dto/
      infrastructure/
        repositories/
        adapters/
      interfaces/
        http/
    signal/
    paper-trading/
```

规则：

1. Controller 只处理路由、鉴权、DTO 校验和响应状态。
2. Use Case / Application Service 编排用例、事务边界和跨领域协作。
3. Domain 处理核心业务规则、状态转换、领域校验和领域事件产生。
4. Repository 只负责数据访问。
5. Adapter 只负责外部服务适配。
6. DTO、Entity、Repository、Use Case、Domain Service 不得互相越界引用。
7. 核心业务领域采用轻量 DDD 分层：`domain`、`application`、`infrastructure`、`interfaces`。
8. 简单模块可以适度简化，但不得把交易、策略、模拟盘、风控核心逻辑写进 Controller。

## 4. 命名规范

### 4.1 通用命名

1. 文件名使用 kebab-case。
2. React 组件使用 PascalCase。
3. 变量和函数使用 camelCase。
4. 类型、接口、类使用 PascalCase。
5. 常量使用 UPPER_SNAKE_CASE。
6. 数据库表和字段使用 snake_case。
7. API 路径使用 kebab-case。

### 4.2 业务命名

必须明确区分模拟与真实：

1. 模拟盘使用 `paper`、`paperTrading`、`paperAccount`、`paperOrder`、`paperTrade`。
2. 真实交易预留使用 `real`、`exchange`、`automation`，且 MVP 禁用。
3. 不得用 `order`、`trade`、`position` 泛称模拟交易对象，避免混淆。
4. 用户可见文案必须写“模拟盘”“模拟订单”“模拟成交”。

## 5. TypeScript 与类型规范

1. 开启 TypeScript strict 模式。
2. 禁止使用 `any`，除非有注释说明第三方边界原因。
3. API 请求和响应必须定义类型。
4. 表单数据必须有 schema 校验。
5. 后端 DTO 必须校验输入。
6. 枚举值必须统一来源，不得在多个文件散落字符串。
7. 金额、价格、比例、手续费不得使用不受控浮点数作为最终计算口径。
8. 日期时间统一使用 ISO 字符串或明确 Date 类型，并约定时区。

### 5.1 文案与多语言预留

1. 用户可见文案默认使用简体中文。
2. QuantFlow 品牌名、币种符号、技术缩写、API 字段和代码标识可以保留英文。
3. 长文案、风险提示、空态说明、错误提示和通知模板应集中管理，不得散落硬编码在多个组件中。
4. 标准免责声明必须复用统一常量。
5. 后续 i18n 应以简体中文作为默认 locale。
6. 禁止在中文页面中大量混用英文营销文案。

## 6. 函数与组件规范

### 6.1 React 组件

1. 组件只接收必要 props。
2. 复杂组件拆成容器组件和展示组件。
3. 表单逻辑、请求逻辑、权限逻辑应提取 hook。
4. 图表组件必须支持 loading、empty、error、sample-insufficient 状态。
5. 策略、信号、模拟盘组件必须同时展示收益和风险。
6. 不得在组件中硬编码会员权限、风险等级或禁止文案词库。

### 6.2 函数

1. 函数名应表达业务意图。
2. 函数应单一职责。
3. 复杂布尔判断必须提取为具名函数。
4. 纯计算逻辑应保持无副作用，便于测试。
5. 涉及风险、权限、会员权益的函数必须有单元测试。

## 7. 后端分层规范

1. 后端采用轻量 DDD + 模块化单体。
2. 按业务领域拆分模块，不按技术层随意堆 service。
3. 核心领域模块区分 `domain`、`application`、`infrastructure`、`interfaces`。
4. Controller 不写业务逻辑。
5. Use Case / Application Service 负责编排流程。
6. Domain 负责核心规则、状态转换和领域事件。
7. Repository 不做权限判断或业务判断。
8. Domain 不依赖 HTTP 请求对象、Prisma、Nest request 或外部 adapter。
9. 风控规则独立于 Controller 和前端。
10. 会员权益校验必须在后端执行。
11. 管理端 RBAC 必须在后端执行。
12. 后续真实交易模块不得注册 MVP 可访问路由。
13. 不使用微服务、复杂 CQRS 或 Event Sourcing。
14. 允许领域事件，例如 `StrategySignalGenerated`、`PaperOrderExecuted`、`RiskEventTriggered`；第一版使用本地事件或 outbox/worker，不引入复杂事件总线。

## 8. 错误处理与日志规范

### 8.1 错误处理

1. 不允许吞掉异常。
2. 用户可见错误必须使用明确文案。
3. 内部错误不得暴露密钥、token、SQL、堆栈细节。
4. 业务错误使用稳定错误码。
5. 权限不足、会员不足、策略不可用、信号过期、模拟盘超限必须区分错误码。

### 8.2 日志

必须记录：

1. 用户登录异常。
2. 策略状态变更。
3. 信号取消或异常标记。
4. 模拟盘暂停、恢复、异常。
5. 会员手动开通、延长和取消。
6. 权限变更。
7. 风险事件处理。

禁止记录：

1. 邮箱验证码及其明文、Resend API Key。
2. session token 或完整 Cookie。
3. 支付敏感信息。
4. 后续版本中的 API Secret。

## 9. 配置与环境变量规范

1. 所有环境差异通过环境变量或配置文件管理。
2. 不得在代码中硬编码密钥、支付配置、行情源密钥或管理员账号。
3. 必须提供 `.env.example`，只包含示例值。
4. feature flag 必须默认关闭后续交易能力。
5. 配置读取必须集中封装并做启动时校验。

MVP 配置层必须默认关闭并在服务端启动时校验。生产会员支付是例外：`enableProductionPayments` 默认关闭，但生产环境可显式开启 Plisio 会员支付；不得因此开放任何实盘交易能力。

1. `enableExchangeConnection`。
2. `enableSemiAutoTrading`。
3. `enableAutoTrading`。
4. `enableAuthorPortal`。
5. `enableProductionPayments`。

## 10. 测试组织规范

### 10.1 测试文件

1. 单元测试与被测文件邻近或放在业务域 `__tests__` 目录。
2. E2E 测试按用户路径组织。
3. 测试命名必须描述业务行为。
4. 避免只测试实现细节。

### 10.2 必测场景

1. 会员权益限制。
2. 策略状态限制。
3. 信号过期和取消。
4. 模拟盘数量限制。
5. 模拟盘风控拦截。
6. 收益与风险并列展示。
7. 风险免责声明展示。
8. 管理端 RBAC。
9. 审计日志。
10. 后续真实交易能力不可访问。

## 11. 禁止事项

1. 禁止大文件堆功能。
2. 禁止在页面组件里直接写复杂业务规则。
3. 禁止复制粘贴相同逻辑超过 3 次。
4. 禁止用魔法字符串表示策略状态、信号状态、会员等级、风险等级。
5. 禁止把模拟交易命名为真实交易。
6. 禁止在 MVP 暴露真实交易入口。
7. 禁止吞异常或只打印不处理。
8. 禁止绕过后端权限校验。
9. 禁止在代码中硬编码密钥。
10. 禁止新增功能但不更新相关文档。
11. 禁止把交易、策略、模拟盘、风控核心逻辑写在 Controller。
12. 禁止在 MVP 引入微服务、复杂 CQRS、Event Sourcing 或复杂事件总线。

## 12. 验收标准

1. 单个源码文件默认不超过 300 行，超过 400 行必须拆分或属于明确例外。
2. 单个函数默认不超过 60 行，复杂规则已拆分为具名函数或领域服务。
3. 前端页面、组件、hook、API、权限、格式化逻辑分层清晰。
4. 后端 Controller、Use Case、Domain、Repository、Adapter 边界清晰。
5. 命名清晰区分模拟交易与未来真实交易。
6. TypeScript 类型、DTO、schema 和错误码完整。
7. 权限、会员权益、风控和风险提示有测试覆盖。
8. 新增或变更范围已同步更新相关文档。
9. 核心领域模块符合轻量 DDD 分层，简单模块未过度抽象。
