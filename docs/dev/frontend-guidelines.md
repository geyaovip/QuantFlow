## 1. 文档目标

本文档定义 QuantFlow MVP 前端开发规范，确保官网、应用工作台、管理后台、组件库和接口调用保持一致。

MVP 默认浅色主题。样式必须通过语义 design tokens / CSS variables 实现；不提供深色主题切换，不为深色维护独立组件分支。

QuantFlow MVP 前端不得展示真实交易所 API 连接、真实下单、半自动交易或全自动交易入口。只保留全局禁用 feature flag，不创建对应路由、权限或组件动作。

## 2. 技术栈

官网、应用工作台和管理端统一使用：

1. Next.js 16.2 + React 19.2 + TypeScript 6 strict。
2. Tailwind CSS 4 + `packages/ui` v0 design tokens。
3. TanStack Query 5。
4. Zod 4。
5. 优先 React 状态与 URL 状态；确有跨页面客户端状态再选择轻量状态库。
6. ECharts 6 和 Lucide React。

不引入 Ant Design，避免其视觉和组件体系与 QuantFlow v0 tokens 并存。完整版本、测试和 monorepo 结构见 `technical-baseline.md`。

## 3. 项目结构

推荐结构：

```text
src/
  app/
  components/
    base/
    business/
    charts/
    admin/
    layouts/
  features/
    auth/
    strategies/
    signals/
    paper-trading/
    membership/
    notifications/
    admin/
  lib/
    api/
    auth/
    permissions/
    formatters/
    validators/
  styles/
    tokens.css
    globals.css
  types/
```

模块说明：

1. `components/base` 优先存放 Button、Card、Badge、RiskBadge、Modal/Dialog 和通用状态组件。
2. `components/business` 优先存放 MetricCard、StrategyCard、SignalCard、DataTable、ChartCard 和 PageHeader。
3. `components/layouts` 存放 MarketingLayout、UserAppLayout、AdminLayout 和必要的 SidebarLayout。
4. `features` 按业务域组织页面、hooks、请求和局部状态。
5. `lib/api` 统一封装请求、错误处理、鉴权头和响应解析。
6. `styles/tokens.css` 定义 `ui-guidelines.md` 的 v0 颜色与尺寸语义 token；页面不得硬编码主题色。
7. 排版、间距、圆角、阴影、控件高度、焦点和动效复用 CSS tokens；断点复用构建时主题配置或共享常量，不在组件局部创建第二套尺度。

## 4. 代码组织与质量约束

详细规则以 `docs/dev/code-organization-guidelines.md` 为准。

前端必须遵守：

1. 单个源码文件建议不超过 300 行，超过 400 行必须拆分。
2. React 页面文件建议不超过 250 行。
3. React 组件文件建议不超过 200 行。
4. 自定义 hook 文件建议不超过 150 行。
5. 单个函数建议不超过 60 行。
6. 页面只做布局和模块组合，不直接堆复杂业务规则。
7. 请求逻辑、权限判断、格式化、风险规则、会员权益判断必须拆到独立模块。
8. 策略、信号、模拟盘组件不得只展示收益，必须展示风险配套指标。

必须拆分的情况：

1. 一个页面同时包含列表、详情、表单和弹窗的大量逻辑。
2. 一个组件同时处理 API 请求、权限判断、表单校验和图表展示。
3. 同一格式化、权限或风险判断逻辑复制 3 次以上。
4. 模拟交易命名与真实交易命名开始混淆。

## 5. 页面开发规范

### 5.1 官网页面

路径：`/`。

必须实现：

1. 官网首页。
2. 产品定位。
3. 核心功能。
4. 策略信号预览。
5. 模拟盘说明。
6. 风控理念。
7. 定价与会员权益说明。
8. 风险提示。
9. 右上角“进入应用” CTA。

页面规则：

1. 官网负责品牌、信任、转化和风险教育。
2. “进入应用”未登录时点击进入 `/login?next=/app/strategies`，登录成功后进入 `/app/strategies`。
3. `/app` 必须重定向到 `/app/strategies`。
4. 官网可以更轻量、更有营销表达，但不能出现稳赚、保本、躺赚、零风险等表述。
5. 官网不承载完整策略筛选、模拟盘管理或后台操作。

### 5.2 应用工作台页面

路径前缀：`/app`。

必须实现：

1. `/app` 必须重定向到 `/app/strategies`。
2. 用户端邮箱验证码登录；不提供密码注册和忘记密码页面。
3. `/app/strategies` 策略广场。
4. `/app/strategies/:id` 策略详情。
5. `/app/signals` 信号中心。
6. `/app/my-strategies` 我的策略。
7. `/app/paper-trading` 模拟盘列表、创建、详情。
8. `/app/membership` 会员计划与当前订阅；生产支付开启时创建 Plisio invoice 并跳转支付页，支付结果以后端回调为准。
9. `/app/profile` 个人中心。

页面规则：

1. 策略和信号页面必须展示风险提示。
2. 收益指标必须同时展示风险指标。
3. 创建模拟盘前必须展示模拟盘说明，不得让用户误认为实盘。
4. 未登录访问受保护页面时跳转登录并保留返回地址。
5. 应用工作台使用浅灰页面背景、白色内容卡片、轻边框和极少阴影。
6. 应用工作台更强调数据、策略、信号、模拟盘和风险并列展示。

### 5.3 管理端页面

路径前缀：`/admin`。

必须实现：

1. 管理端邮箱验证码登录；仅已授权管理员可进入。
2. 数据看板。
3. 用户管理。
4. 策略管理。
5. 信号管理。
6. 模拟盘管理。
7. 会员管理。
8. 风险管理。
9. 权限管理。
10. 系统日志。

页面规则：

1. 所有敏感操作必须二次确认。
2. 所有管理端页面必须校验管理员权限。
3. 表格字段必须支持空值、异常状态和权限隐藏。
4. 管理端使用侧边栏、白色内容区和高密度表格；主操作为深灰/黑色。
5. 管理端更强调高密度表格、筛选、状态管理和操作效率。

### 5.4 自适应页面

1. 所有页面必须适配窄屏、中等宽度和宽屏，使用 `ui-guidelines.md` 的 640/768/1024/1280/1536px 断点。
2. 优先使用 CSS Grid、Flexbox、`minmax()`、容器约束和内容驱动换行，禁止按设备型号堆媒体查询。
3. 官网模块更轻量，应用工作台卡片自动换列；管理端侧边栏在窄屏折叠，DataTable 保留关键列并提供横向滚动或详情入口。
4. 弹窗宽度受视口约束，表单窄屏单列，图表自适应容器且保留最小可读高度。
5. 不得隐藏风险指标、权限反馈、错误信息或关键操作来“解决”小屏布局。

### 5.5 移动端导航

官网 `/`：

1. 移动端可以使用顶部导航 + 汉堡菜单。
2. “进入应用”仍必须易发现、易点击。
3. 汉堡菜单只用于官网锚点和辅助链接，不进入应用工作台主导航职责。

应用工作台 `/app`：

1. 移动端必须使用底部导航栏作为主导航。
2. 不使用左上角汉堡菜单作为主导航。
3. 顶部区域用于当前页面标题、返回、通知、搜索等辅助操作。
4. 左上角不作为主功能入口，最多用于 Logo 或返回按钮。
5. 底部导航固定为策略、信号、模拟盘、我的，并适配 `safe-area-inset-bottom`。
6. 页面主操作按钮、表单提交按钮、风险提示和 Toast 不得被底部导航遮挡。
7. 页面容器底部必须预留底部导航高度和安全区间距。

管理端 `/admin`：

1. 管理端以桌面端为主。
2. 移动端不作为核心使用场景，但所有必要查看和管理操作仍必须可用。
3. 不要求管理端底部导航；侧边栏折叠为可访问抽屉，表格提供横向滚动或详情视图。

## 6. 状态管理与数据请求

### 6.1 请求规范

1. API base path 使用 `/api/v1` 或环境变量配置。
2. 所有请求通过统一 client 发起。
3. 统一处理 401、403、404、422、429、500。
4. 所有列表使用服务端编号分页、筛选、排序；用户默认 20 条、管理端默认 50 条，最大 100 条，MVP 禁止无限滚动。
5. Mutation 成功后按业务域刷新 query。

### 6.2 缓存策略

1. 策略列表可短缓存。
2. 信号列表应设置较短 stale time。
3. 模拟盘详情应支持手动刷新和定时刷新。
4. 管理端看板允许 5-15 分钟数据延迟。

### 6.3 实时能力

MVP 可先使用轮询，后续接入 WebSocket。

实时场景：

1. 新信号。
2. 模拟盘权益更新。
3. 风险事件。
4. 站内通知。

## 7. 权限与功能开关

### 7.1 用户权限

前端必须识别：

1. Anonymous。
2. Free。
3. Pro。
4. Premium。
5. Admin。

### 7.2 功能开关

建议定义：

1. `enableExchangeConnection`：MVP 默认为 false。
2. `enableSemiAutoTrading`：MVP 默认为 false。
3. `enableAutoTrading`：MVP 默认为 false。
4. `enableAuthorPortal`：MVP 默认为 false。
5. `enableProductionPayments`：默认 false；生产环境可开启 Plisio 会员支付。

规则：

1. false 时不渲染入口、按钮和路由。
2. 不得仅靠前端隐藏保护后端能力，后端也必须禁用。

## 8. 表单与校验

### 8.1 通用规则

1. 必填项清晰标记。
2. 错误信息贴近字段展示。
3. 金额、比例、价格使用数值组件和精度控制。
4. 提交按钮必须有 loading 状态。
5. 验证码输入支持粘贴、自动聚焦、重发倒计时和可访问错误提示；前端倒计时不替代服务端限流。

### 8.2 模拟盘创建表单

字段：

1. 策略。
2. 币种。
3. 初始模拟资金。
4. 单笔最大仓位。
5. 最大持仓数量。
6. 止损规则。
7. 止盈规则。

校验：

1. 初始模拟资金必须大于 0。
2. 单笔最大仓位默认不超过 10%。
3. MVP 杠杆固定为 1 倍或不展示杠杆配置。
4. 提交前展示“不承诺收益、仅为模拟验证”的确认提示。

## 9. 图表与数据展示

1. 收益曲线必须配套回撤曲线。
2. 回测、模拟、实时信号数据必须用图例区分。
3. 所有百分比保留统一精度。
4. 价格和金额按币种精度格式化。
5. 数据不足、加载失败、行情延迟必须有明确状态。
6. 不得用排行榜或视觉动效夸张暗示收益确定性。
7. 图表基础使用中性灰，低饱和蓝仅作有限辅助序列；红、绿、琥珀色只表达数据语义。

## 10. 语言与多语言预留

### 10.1 MVP 默认语言

QuantFlow MVP 面向中国用户，前端用户可见内容默认使用简体中文，包括：

1. 页面标题。
2. 导航。
3. 按钮。
4. 表单 label 和 placeholder。
5. 校验错误。
6. 空态、错误态、加载态。
7. 风险提示。
8. 通知内容。
9. 管理后台操作文案。

可保留英文的内容：

1. QuantFlow 品牌名。
2. BTC、ETH、SOL、USDT 等币种和交易对。
3. API、KYC、P&L 等行业通用缩写，必要时补充中文解释。
4. 代码标识、接口字段、日志 key。

### 10.2 文案管理

1. MVP 可以先使用中文常量文件管理文案。
2. 不建议把长文案、风险提示、空态说明散落硬编码在组件中。
3. 标准免责声明应复用统一常量。
4. 禁止词库和风险提示文案应集中维护。
5. 后续引入 i18n 时，中文文案应作为默认 locale。

### 10.3 多语言预留

后续多语言应考虑：

1. 文案 key 命名。
2. 日期、时间、数字、百分比、币种格式。
3. 时区。
4. 风险免责声明翻译审核。
5. 图表 tooltip 与表格列名本地化。

MVP 不因多语言预留牺牲中文体验，也不要求在 `v1.0.0 MVP Core` 前上线多语言。

## 11. 安全与性能

### 11.1 安全

1. 防止 XSS，富文本内容必须经过白名单处理。
2. 管理端页面必须校验权限。
3. 敏感 token 不得存储在不安全位置。
4. CSRF 防护按后端鉴权方案配合实现。

### 11.2 性能

1. 首页首屏加载目标小于 3 秒。
2. 策略列表加载目标小于 3 秒。
3. 图表组件按需加载。
4. 管理端大表格使用分页，不一次性加载全部数据。
5. 静态资源开启压缩和缓存。

## 12. 验收标准

1. 前端页面覆盖 MVP P0 范围。
2. 真实交易所连接、真实下单、半自动交易和全自动交易 feature flag 默认关闭且无 UI 入口。
3. 收益指标组件强制展示风险配套信息。
4. 策略、信号、模拟盘、会员和管理端页面均有清晰权限处理。
5. 表单、错误态、空态、加载态和权限不足状态可被测试覆盖。
6. 页面、组件、hook、API、权限和格式化逻辑拆分符合 `code-organization-guidelines.md`。
7. 用户可见页面默认使用简体中文，标准免责声明和风险提示使用统一中文文案常量。
8. 默认浅色主题通过视觉回归；不存在深色切换入口、大面积高饱和蓝或重阴影。
9. 用户端与管理端分别符合轻卡片和高密度表格布局，且共用语义 design tokens。
10. Button、Card、Badge、RiskBadge、MetricCard、StrategyCard、SignalCard、DataTable、ChartCard、状态组件、Dialog、PageHeader 和三类 Layout 优先完成。
11. 所有页面在窄屏、中等宽度和宽屏下无不可访问溢出、遮挡或关键内容缺失。
12. `/app` 移动端主导航使用底部导航栏，官网移动端可使用汉堡菜单，管理端移动端保留必要查看和管理操作。
